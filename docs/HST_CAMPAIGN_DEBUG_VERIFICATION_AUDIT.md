# HST Campaign Debug Verification Audit

Date: 2026-07-03


Verdict: not fully implemented.

The current `Run Campaign Debug` implementation is a useful certification scaffold. It is no longer just a string/report smoke runner: it has typed results, deterministic debug run prefixes, artifacts, status/cancel/cleanup commands, stronger bootstrap/preflight/HQ/economy assertions, convoy readiness/progress probing, mission primitive action probes, a POW captive free/follow/extraction probe, seeded persistence smoke roundtrip assertions, a render-bubble zone activation probe, town support up/down mutation assertions, civilian runtime population and bounded movement-sample assertions, QRF/search support route sampling/arrival assertions, Phase 24 pacing/escalation/end assertions, and mission cleanup checks.

It does not yet satisfy the full pasted contract for a complete one-button in-game verification suite. Large areas remain legacy string-wrapped smoke checks or are not represented at all.

## Implemented Evidence

- Admin command wiring exists for start/status/cancel/cleanup in `Scripts/Game/HST/Services/HST_CommandUIService.c`.
- The coordinator exposes `RequestAdminRunCampaignDebug`, `RequestAdminCampaignDebugStatus`, `RequestAdminCancelCampaignDebug`, and `RequestAdminCleanupCampaignDebug`.
- The runner remains sequenced through bootstrap, baseline, HQ, economy/support, early mechanics, mission sweep, phase smoke, final report, and completion.
- Run profiles are accepted as `smoke`, `physical`, and `full`: `smoke` skips the long early/mission/phase sweeps, `physical` keeps early mechanics plus the mission/physical sweep and skips late phase smoke, and `full` preserves the complete sequence.
- Typed result classes exist in `Scripts/Game/HST/Data/HST_CampaignDebugResult.c`.
- Artifacts write to `$profile:h-istasi/debug` as JSON, summary text, and state-diff text.
- Runs now record a deterministic debug marker prefix, mission prefix, and entity tag in JSON, summary, state-diff, and status output.
- Run start removes stale `hst_debug_`-prefixed persisted state before capturing start counts. Forced debug-started missions are retagged before objective/runtime initialization so derived objective/runtime/asset/group/marker records inherit the run prefix.
- Debug-created player support, Phase 19 support smoke records, and Phase 17/18/22 enemy-order seeds are retagged with the current run prefix before marker refresh/report output; support request cases assert the prefix directly.
- Bootstrap records typed assertions for server authority, debug actor access, active campaign repair, HQ state, Petros state, teleport, and player presence.
- Preflight records typed assertions for key services, mission registry count/uniqueness/runtime/duration, compatible debug target zones, default faction/civilian prefab resource resolution, runtime-selected mission prop/vehicle prefab resolution, runtime waypoint prefab resolution, zone graph counts, and physical-war setting.

The income case seeds a resistance-owned income zone, calculates expected money/HR through the town service, asserts exact money/HR deltas, records enemy-owned income potential as excluded, and checks that the force-income command does not unexpectedly mutate the passive timer.

## Not Fully Implemented

- Debug cleanup is deterministic for persisted state, marker records, and physical world entities that were explicitly named with the campaign-debug tag or `hst_debug_` prefix. It still cannot remove arbitrary debug leftovers that were spawned without a queryable debug name/tag.
- Stage 3 support/civilian/undercover coverage is still partial: it now asserts request records, controlled ETA progression, QRF/search physicalization, repeated QRF/search route samples with target-distance closure, stall evidence, and hard no-progress timeout failure, controlled support arrival state, support cancellation, town aid deltas, Phase 20 support up/down mutation and owner flips, physical civilian population counts/faction/position/cleanup, bounded civilian movement samples, and undercover phase transitions, but it does not prove contact/combat behavior after arrival or long-window civilian AI reaction behavior. Current static civilian runtime movement is reported as WARN evidence rather than PASS.
- Early mechanics report/action wrappers now have typed case coverage for the named Phase 0-13 checks: persistence, generated content, render-bubble zone activation, foundation/checkpoint, mission-runtime visibility, primitive runtime sampling, garrison recruit/remove, civilian aid, support cancellation, garage/vehicle/loadout actions, command UI coverage, and captive follow. Remaining early-mechanics limitations are physical-runtime depth gaps covered by the convoy, mission, captive, render-bubble, and persistence bullets below.
- The all-mission sweep now has typed start/runtime/cleanup coverage and exercises primitive-specific runtime commands for `kill_hvt`, `destroy_target`, `recover_cargo`, `deliver_supplies`, `rescue_extract`, `hold_area`, and `clear_area`, but physical behavior coverage is still incomplete. Transport primitives now arrange and clean a temporary carrier, but they still do not prove natural player driving/path travel between pickup and delivery. Area primitives now prove player-radius/runtime-tick objective completion with controlled hostile absence, but they still do not prove observed hostile combat or natural area-clearing behavior. Admin completion remains the fallback cleanup path for missions that do not complete through the primitive probe.
- Convoy probing now forces debug convoy departure windows, waits over multiple five-second progress intervals, records repeated sample counts, distance-closed metrics, recovery attempts, sampled travel/contact phase history, and hard-fails fully sampled no-movement/no-distance-closure convoy windows that do not record recovery or contact/terminal phase progress. It still does not prove the full phase history from staging to moving to contact to arrival/elimination for every convoy mission.
- The POW/captive probe now proves free/follow state, controlled repeated follow sample state with hard no-closure timeout evidence, all-required-captive extraction completion, alive captive count, and exact money/HR deltas. It still does not prove boarding/transport behavior.
- Phase 14-24 smoke steps now use typed phase cases for pass/fail classification instead of duplicate legacy string-result rows. Their remaining gaps are physical-depth gaps, not phase-smoke result wrapping: Defend Petros multi-wave/contact/arrival behavior still needs stronger typed assertions; Phase 16 proves abstract garrison/training state only and does not physicalize recruited units; Phase 17 samples one physical counterattack support group but does not prove multi-wave/contact/arrival/resolution behavior; Phase 20 proves support up/down state mutation, support-only owner flips, and zone marker model reflection, but not long-running civilian reaction behavior or rendered map widgets; Phase 22 samples one physical Petros attacker group but does not prove multi-wave/contact/arrival/defense pressure behavior; Phase 23 proves command/marker model state, negative-path no-mutation, and native report/purge availability, but does not visually inspect rendered map widgets; Phase 24 proves controlled escalation ticks, but not long-window autonomous campaign pressure over many normal campaign loops.
- Background war and commander target testing is folded into the Phase 18 typed smoke step rather than implemented as a separate stage. It verifies command selection/resource spending for one controlled tick, but it does not prove physical group movement/arrival or long-running occupier-vs-invader campaign dynamics.
- Quantitative war level escalation/aggression pressure tests now run over controlled Phase 24 windows. Remaining gaps are long-window autonomous escalation frequency, heavier support eligibility across varied POI contexts, and natural occupier-vs-invader pressure over many normal campaign loops.
- Counterattack physical spawn/advance is now sampled across repeated route windows through Phase 17's real enemy-order/support/active-group route path with hard no-progress timeout failure, but counterattack wave/contact/arrival/resolution behavior is not yet implemented as a typed test.
- Player render-bubble coverage now proves one clean zone far/near/leave activation and repeated cleanup timeout path through the real physical-war update. It still does not cover mission-asset bubble policy, convoy-expired behavior, visual/rendered inspection, or repeated movement windows across multiple zone types.
- Persistence smoke now seeds and asserts active mission, convoy, primitive, garage/cargo, support/order, civilian, and undercover records through an in-memory save-data restore path. It still does not perform a real process restart, multiclient reconnect/soak, or a broader field-vehicle restore outside the seeded smoke records.
- Final artifacts include the requested summary matrix sections, but those matrices still expose partial coverage because some bootstrap/HQ/baseline/final records remain legacy string wrappers.
- Cleanup leak checks now run after non-legacy typed cases, and Phase 14-24 plus mission-sweep start/runtime/cleanup cases are no longer duplicated by legacy result rows. Remaining bootstrap/HQ/baseline/final legacy string-wrapped cases still need migration before every major case has equivalent cleanup evidence. Prefixed persisted state cleanup now includes a conservative world scan for entities named with the debug tag/prefix, but cleanup still depends on debug spawn paths naming their physical entities.
- Stall detection and timeout evidence dumps are implemented for Stage 3 ground support, convoy movement windows, captive follow windows, render-bubble leave cleanup, Phase 17/22 enemy-order route probes, and Phase 20 civilian movement samples. They are still not implemented uniformly for every physical test category.
